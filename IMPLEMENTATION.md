## MILESTONE 1 — TERMINAL PANEL

### Definition of Done

- [x] TerminalPanel component renders inside a PanelContainer
- [x] Real PowerShell process spawned via Tauri shell command on panel mount
- [x] xterm.js renders the shell output correctly
- [x] User can type commands and receive responses (full PTY interaction)
- [x] Shell process is killed cleanly on panel unmount (no orphaned processes)
- [x] Terminal uses projectRoot as working directory from PanelProps
- [x] Terminal renders with correct theme colors from CSS variables
- [x] PTY is viable on Windows — portable-pty (ConPTY) compiles and works

### Why This Milestone Is First

The terminal panel is the highest-risk integration in the stack. xterm.js + Tauri PTY
on Windows involves native OS process management, IPC between Rust and JavaScript,
and third-party library compatibility. If this integration has fundamental problems,
we need to know in week one — not after the entire UI is built around it.

Nothing else is built until this works end to end.

### Ordered Task List

1. Install xterm.js and required addons:
   `pnpm add @xterm/xterm @xterm/addon-fit @xterm/addon-web-links`
2. Add Tauri shell plugin to Cargo.toml and tauri.conf.json
3. Write Rust command in `src-tauri/src/commands/shell.rs`:
   - `spawn_shell(cwd: String, shell: String) -> Result<String, String>`
     Returns a PTY process ID
   - `write_to_shell(pty_id: String, data: String) -> Result<(), String>`
   - `kill_shell(pty_id: String) -> Result<(), String>`
4. Register commands in `src-tauri/src/main.rs`
5. Write TypeScript shell wrappers in `src/shell/terminal.ts`
6. Write TerminalPanel component in `src/panels/terminal/TerminalPanel.tsx`:
   - On mount: call spawnShell with cwd from settings.cwd or projectRoot
   - Initialize xterm.js Terminal instance
   - Wire xterm onData → writeToShell
   - Wire shell output events → xterm.write
   - Apply FitAddon to fill panel container
   - After shell spawns: execute settings.startupCommands sequentially
     Each command is written to the shell with a newline — fired in order
   - On unmount: call killShell, dispose xterm instance
7. Write PanelSkeleton component (loading state for all panels)
8. Write minimal PanelContainer with Suspense + PanelSkeleton
9. Write a temporary hardcoded test layout in App.tsx:
   - Single PanelContainer rendering TerminalPanel
   - Hardcoded settings: shell='powershell.exe', cwd='C:/users/patri'
10. Test end to end: app renders, PowerShell starts, commands work, unmount cleans up
11. Document any PTY issues or workarounds in ERRORS.md
12. Commit: "M1 complete — terminal panel end to end"

### Notes

The temporary test layout in App.tsx from step 9 is scaffolding only.
It will be replaced entirely in M2. Do not build UI around it.

If full PTY is not achievable, the fallback is a command-execution model:
user types a command, it runs via Tauri shell, output is captured and displayed.
This is a degraded but functional terminal experience. Document the decision
in DECISIONS.md with full rationale before implementing the fallback.

## APPENDIX B — BUILD ORDER RATIONALE

The milestone order is not arbitrary. It is risk-ordered:

**M0 before everything** — A bad scaffold costs hours. A bad scaffold discovered
in M4 costs weeks.

**M1 before UI** — Terminal is the highest-risk integration. If xterm.js + Tauri PTY
on Windows has fundamental issues, know it before building the shell around it.

**M2 before persistence** — Build the UI shell with hardcoded data first. Wiring
persistence to broken UI wastes time. Confirm the UI is right, then persist it.

**M3 before panels** — Persistence must work before panels are wired to real data.
A panel that reads a projectRoot that isn't persisted correctly is debugging two
things at once.

**M4 after persistence** — Real panels need real project roots. Persistence must
be reliable before panels depend on it.

**M5 after panels** — Widgets read from the same data sources as panels. The
Tauri commands for git and filesystem are written in M4 — M5 reuses them.

**M6 last** — Polish on unstable foundation is wasted effort. Polish only after
all features are working.
